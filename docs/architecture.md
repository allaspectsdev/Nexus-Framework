# Architecture Deep Dive

## The Query Loop State Machine

The core of Nexus is `engine/QueryLoop.ts` — a flat `while(true)` loop that manages the entire lifecycle of an LLM conversation with tool use.

### Why a Flat Loop?

Recursive agent designs (where each tool result triggers a new function call) have three problems:
1. Stack depth grows with conversation length — eventual overflow on long sessions
2. Error recovery requires unwinding multiple frames
3. Audit trails are scattered across call frames

The flat loop solves all three: state is an explicit object passed through iterations, transitions are tracked, and recovery is just `continue` with modified state.

### Loop Phases

Each iteration follows this sequence:

```
1. Pre-flight checks
   ├── Abort signal?  → Terminal(aborted)
   ├── Max turns?     → Terminal(max_turns)
   └── Continue

2. Route to model
   └── Router classifies → local or Claude

3. Stream from model
   ├── Accumulate text, thinking, tool_use blocks
   ├── Yield StreamEvents to caller
   └── Capture final message + stop_reason + usage

4. Handle stop reason
   ├── end_turn      → Terminal(completed)
   ├── max_tokens    → Escalate 8K→64K, then inject resume → Continue
   └── tool_use      → Execute tools → Continue

5. Tool execution
   ├── Validate inputs via Zod
   ├── Execute (concurrent read-only, serial writes)
   └── Append tool_result messages

6. Build next state
   └── state = { ...state, messages, turnCount++ }
```

### State Transitions

Every `continue` carries a reason:

| Transition | Meaning |
|---|---|
| `tool_use` | Model requested tools; results appended |
| `max_tokens_escalation` | First max_tokens hit; retrying at 64K |
| `max_tokens_recovery` | Injecting "resume directly" message |
| `compact_retry` | Context was compacted; retrying |
| `next_turn` | Normal continuation |

### The AsyncGenerator Pattern

`queryLoop()` is an `AsyncGenerator<StreamEvent, Terminal>`. This means:
- The caller iterates with `for await (const event of loop)` to receive streaming events
- The generator `return`s a `Terminal` when done
- The caller can break early to abort

This is the same pattern used in production agentic systems for composable streaming.

## Hybrid Model Routing

### The Router Decision Process

```
                    ┌─────────────┐
                    │ User Message│
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
                    │  Classify   │
                    │  Complexity │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │
         < threshold  = threshold  > threshold
              │            │            │
         ┌────▼───┐   ┌───▼───┐   ┌───▼────┐
         │ Local  │   │ Local │   │ Claude │
         │ Model  │   │(if up)│   │  API   │
         └────────┘   └───────┘   └────────┘
```

### Complexity Signals

The router uses pattern matching on the user's message:

- **High complexity** (→ Claude): `implement`, `refactor`, `debug`, `why`, `explain how`
- **Low complexity** (→ local): `summarize`, `list`, `format`, `extract`, `count`
- **Heuristic fallback**: message length (longer = more complex)

### The ModelClient Interface

Both Claude and local clients implement the same interface:

```typescript
type ModelClient = {
  readonly provider: 'local' | 'claude'
  stream(options: ModelCallOptions): AsyncGenerator<StreamEvent>
  isAvailable(): Promise<boolean>
}
```

This makes routing transparent — the QueryLoop doesn't know or care which model is responding.

## Context Management

### Tiered Decay

The fundamental insight: most tool results are only useful for 2-3 turns after they're generated. A `git status` from 10 turns ago is stale. A file read from 15 turns ago might have been edited since.

```
Turn 0   [████████████████] Full result (5K tokens)
Turn 3   [████░░░░░░░░░░░░] Summary (500 tokens) — generated by local model
Turn 6   [█░░░░░░░░░░░░░░░] Stub (50 tokens) — "[Tool result cleared]"
```

The local model generates summaries for free. Even if the local model is offline, Nexus falls back to extractive summarization (first 200 chars).

### Incremental Compaction

When context exceeds 80% of the limit, instead of summarizing the entire history (expensive), Nexus compacts the oldest messages while keeping recent ones verbatim:

```
Before: [msg1, msg2, msg3, ..., msg25, msg26, msg27, msg28, msg29, msg30]
After:  [summary(msg1-20), msg21, msg22, msg23, msg24, msg25, msg26, msg27, msg28, msg29, msg30]
```

The summary is generated by the local model — no API cost.

### CacheSafeParams

When forking sub-agents, cache sharing requires byte-identical API request prefixes. `CacheSafeParams` bundles the five fields that form the cache key:

```typescript
type CacheSafeParams = {
  systemPrompt: string      // Must be identical
  tools: ToolSchema[]       // Must be identical
  model: string             // Must be identical
  messagePrefix: Message[]  // Must be identical
  thinkingConfig: object    // Must be identical
}
```

For multiple children spawned from the same parent, they all receive identical `tool_result` placeholder messages. Only the final directive text differs — maximizing the shared cache prefix.

## Multi-Agent Orchestration

### The Coordinator Pattern

```
Coordinator (main QueryLoop)
│
├── spawn_worker("explorer") ──▶ Worker QueryLoop
│                                 ├── ReadFile(package.json)
│                                 ├── Bash(ls -la src/)
│                                 └── return summary
│
├── spawn_worker("analyzer") ──▶ Worker QueryLoop
│                                 ├── Grep(function, *.ts)
│                                 ├── ReadFile(src/index.ts)
│                                 └── return summary
│
├── spawn_worker("reviewer") ──▶ Worker QueryLoop
│                                 ├── Grep(TODO|FIXME, *.ts)
│                                 ├── ReadFile(src/tools/*.ts)
│                                 └── return summary
│
├── [Workers complete in parallel]
│
├── Receive <task-notification> from each worker
│
└── Synthesize into final response
```

### Task Notifications

Workers communicate results via XML-structured messages:

```xml
<task-notification>
  <task-id>agent_a1b2c3d4</task-id>
  <name>explorer</name>
  <status>completed</status>
  <summary>Found 35 TypeScript files across 8 directories...</summary>
  <result>Full detailed findings...</result>
  <usage>
    <total-tokens>12847</total-tokens>
    <tool-uses>8</tool-uses>
    <duration-ms>4521</duration-ms>
  </usage>
</task-notification>
```

### Worker Isolation

Each worker gets:
- Its own message history (clean context)
- Its own AbortController (linked to parent — parent abort kills children)
- Shared tool registry (same tools as coordinator)
- Shared router (same routing decisions)
- Its own agent registry entry (tracked by coordinator)

## Streaming Tool Execution

### Concurrency Model

```
Model streaming: ──────[text]──[tool_use:ReadFile]──[tool_use:Grep]──[tool_use:Bash]──[end]──
Tool execution:                 └──ReadFile──────┘
                                    └──Grep──────────┘
                                                       └──Bash─────────┘ (serial, not concurrent)
```

Read-only tools start immediately as their input JSON completes. Write tools wait in a queue.

### Sibling Abort

If a concurrent tool errors, a sibling `AbortController` cancels other running concurrent tools — but does NOT abort the parent conversation. The error is captured as a `tool_result` with `is_error: true`, and the model can recover.

## Observability Pipeline

```
[Every Component] ──emit──▶ EventBus ──▶ MetricsCollector ──▶ TerminalUI (stderr)
                                    │                      └──▶ Dashboard (SSE → browser)
                                    └──▶ Event Log (recent 200)
```

### Event Types

| Event | Data |
|---|---|
| `token_usage` | provider, inputTokens, outputTokens, cachedTokens, cost |
| `routing_decision` | provider, reason, inputTokens |
| `agent_lifecycle` | agentId, name, status, durationMs |
| `tool_execution` | tool, durationMs, concurrent, isError |
| `context_decay` | tier, tokensSaved, messageCount |
| `compaction` | tokensBefore, tokensAfter, tokensSaved |
| `stream_text` | text, provider |

### Savings Calculation

The MetricsCollector computes savings by comparing actual cost against a hypothetical "everything goes to Claude" baseline:

```
hypotheticalCost = (localTokens / 1M) * claudeInputPrice
actualCost = claudeCost + localCost (≈ $0)
savings = hypotheticalCost + contextDecaySavings
```
